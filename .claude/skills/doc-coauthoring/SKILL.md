# 文档协作撰写工作流

本技能提供一套结构化工作流，引导用户完成协作式文档创作。作为主动引导者，带领用户完成三个阶段：背景收集、完善与结构化、读者测试。

## 何时提供此工作流

**触发条件：**
- 用户提到撰写文档："写一份文档"、"起草一个提案"、"创建规格说明"、"写一下"
- 用户提到特定文档类型："PRD"、"设计文档"、"决策文档"、"RFC"
- 用户似乎正在开始一项实质性的写作任务

**初始建议：**
向用户提供结构化的文档协作撰写工作流。说明三个阶段：

1. **背景收集**：用户提供所有相关背景，Claude 提出澄清性问题
2. **完善与结构化**：通过头脑风暴和编辑，逐节构建文档
3. **读者测试**：用全新的 Claude 实例（无上下文）测试文档，在他人阅读前找出盲点

说明此方法有助于确保文档在他人阅读时（包括将其粘贴到 Claude 中时）能够有效传达。询问用户是否希望尝试此工作流，或者倾向于自由撰写。

如果用户拒绝，则自由撰写。如果用户接受，进入第一阶段。

## 第一阶段：背景收集

**目标：** 缩小用户所知与 Claude 所知之间的差距，以便后续提供更智能的指导。

### 初始问题

首先询问用户文档的元背景：

1. 这是什么类型的文档？（例如：技术规格、决策文档、提案）
2. 主要受众是谁？
3. 有人阅读后期望产生什么影响？
4. 是否有需要遵循的模板或特定格式？
5. 还有其他需要了解的约束或背景吗？

告知他们可以用简写或以任何方便的方式提供信息。

**如果用户提供了模板或提到了文档类型：**
- 询问是否有模板文档可以分享
- 如果提供了共享文档链接，使用适当的集成工具获取
- 如果提供了文件，则读取它

**如果用户提到编辑现有共享文档：**
- 使用适当的集成工具读取当前状态
- 检查是否有缺少 alt 文本的图片
- 如果存在没有 alt 文本的图片，说明当他人用 Claude 理解文档时，Claude 将无法看到这些图片。询问是否需要生成 alt 文本。如果需要，请他们将每张图片粘贴到对话中以生成描述性 alt 文本。

### 信息倾倒

初始问题回答完毕后，鼓励用户倾倒所有背景信息。请求以下信息：
- 项目/问题背景
- 相关团队讨论或共享文档
- 为什么不使用替代方案
- 组织背景（团队动态、历史事件、内部关系）
- 时间压力或约束
- 技术架构或依赖
- 利益相关者的关切

建议他们不必担心组织方式——只需把所有内容都说出来。提供多种提供背景的方式：
- 意识流式信息倾倒
- 指向团队频道或讨论串
- 链接到共享文档

**如果有可用集成**（例如 Slack、Teams、Google Drive、SharePoint 或其他 MCP 服务器），提到可以直接用这些工具拉取背景信息。

**如果未检测到集成且在 Claude.ai 或 Claude 应用中：** 建议他们可以在 Claude 设置中启用连接器，以便直接从消息应用和文档存储中提取背景信息。

告知他们在初始倾倒后将会提出澄清性问题。

**在背景收集过程中：**

- 如果用户提到团队频道或共享文档：
  - 如果集成可用：告知将立即读取内容，然后使用适当的集成
  - 如果集成不可用：说明无法访问。建议在 Claude 设置中启用连接器，或直接粘贴相关内容。

- 如果用户提到未知的实体/项目：
  - 询问是否应搜索连接的工具以了解更多信息
  - 等待用户确认后再搜索

- 随着用户提供背景，追踪已了解的内容和仍不清楚的内容

**提出澄清性问题：**

当用户表示已完成初始倾倒（或在提供了大量背景后），提出澄清性问题以确保理解：

根据背景中的空缺生成 5-10 个编号问题。

告知他们可以用简写回答（例如："1：是，2：见 #频道，3：不行因为向后兼容"），链接到更多文档，指向频道，或继续信息倾倒。以对他们最高效的方式为准。

**退出条件：**
当问题显示出已有足够理解时——可以询问边界情况和权衡取舍，而无需解释基础内容时——即可认为背景收集充分。

**过渡：**
询问在此阶段是否还有其他背景要提供，或者是否可以开始起草文档。

如果用户想要补充，让他们继续。准备好后，进入第二阶段。

## 第二阶段：完善与结构化

**目标：** 通过头脑风暴、策划和迭代完善，逐节构建文档。

**向用户说明：**
说明将逐节构建文档。对于每一节：
1. 将提出关于应包含什么内容的澄清性问题
2. 将头脑风暴 5-20 个选项
3. 用户指示保留/删除/合并哪些内容
4. 起草该节
5. 通过精准编辑进行完善

从最多未知点的节开始（通常是核心决策/提案），然后逐步完成其余部分。

**节的排序：**

如果文档结构清晰：
询问他们想从哪一节开始。

建议从最多未知点的节开始。对于决策文档，通常是核心提案。对于规格说明，通常是技术方法。摘要节最好留到最后。

如果用户不知道需要哪些节：
根据文档类型和模板，建议适合该文档类型的 3-5 个节。

询问此结构是否合适，或是否需要调整。

**一旦结构达成一致：**

创建带有所有节占位符文本的初始文档结构。

**如果可以访问工件：**
使用 `create_file` 创建工件。这为 Claude 和用户都提供了一个可以参考的框架。

告知将创建包含所有节占位符的初始结构。

创建包含所有节标题和简短占位符文本（如"[待撰写]"或"[内容待填]"）的工件。

提供框架链接并说明现在开始逐节填充。

**如果无法访问工件：**
在工作目录中创建 Markdown 文件。适当命名（例如 `decision-doc.md`、`technical-spec.md`）。

告知将创建包含所有节占位符的初始结构。

创建包含所有节标题和占位符文本的文件。

确认文件名已创建，说明现在开始逐节填充。

**对于每一节：**

### 步骤 1：澄清性问题

宣布将开始处理 [节名] 节。提出 5-10 个关于应包含什么内容的澄清性问题：

根据背景和节的目的生成 5-10 个具体问题。

告知他们可以用简写回答，或只需指出重要内容。

### 步骤 2：头脑风暴

对于 [节名] 节，根据节的复杂性，头脑风暴 [5-20] 个可能包含的内容。寻找：
- 提供的背景中可能被遗忘的内容
- 尚未提及的角度或考虑因素

根据节的复杂性生成 5-20 个编号选项。最后，如果需要更多选项，提出可以继续头脑风暴。

### 步骤 3：策划

询问应该保留、删除或合并哪些点。请求简短的理由以帮助了解下几节的优先级。

提供示例：
- "保留 1,4,7,9"
- "删除 3（与 1 重复）"
- "删除 6（受众已知）"
- "合并 11 和 12"

**如果用户给出自由格式的反馈**（例如"看起来不错"或"我大部分喜欢，但..."）而不是编号选择，提取他们的偏好并继续。解析他们想要保留/删除/更改的内容并应用。

### 步骤 4：缺口检查

根据他们选择的内容，询问 [节名] 节是否还有遗漏的重要内容。

### 步骤 5：起草

使用 `str_replace` 将此节的占位符文本替换为实际起草的内容。

宣布现在将根据他们选择的内容起草 [节名] 节。

**如果使用工件：**
起草后，提供工件链接。

请他们通读并指出需要更改的地方。注明具体说明有助于了解下几节的写作风格。

**如果使用文件（无工件）：**
起草后，确认完成。

告知 [节名] 节已在 [文件名] 中起草。请他们通读并指出需要更改的地方。注明具体说明有助于了解下几节的写作风格。

**给用户的关键说明（在起草第一节时包含）：**
提供说明：不要直接编辑文档，请指出需要更改的内容。这有助于了解他们的写作风格，以便后续节使用。例如："删除 X 要点——Y 已经涵盖了"或"使第三段更简洁"。

### 步骤 6：迭代完善

随着用户提供反馈：
- 使用 `str_replace` 进行编辑（永远不要重新打印整个文档）
- **如果使用工件：** 每次编辑后提供工件链接
- **如果使用文件：** 仅确认编辑完成
- 如果用户直接编辑文档并要求读取：记录他们做的更改，并在后续节中牢记（这显示了他们的偏好）

**继续迭代**直到用户对该节满意。

### 质量检查

连续 3 次迭代没有实质性更改后，询问是否可以在不丢失重要信息的情况下删除任何内容。

节完成后，确认 [节名] 已完成。询问是否准备好进入下一节。

**对所有节重复此过程。**

### 接近完成

在接近完成时（80% 以上的节已完成），宣布打算重新阅读整个文档并检查：
- 各节之间的流畅性和一致性
- 冗余或矛盾之处
- 任何感觉像"模板填充"或通用填充文本的内容
- 每个句子是否都有价值

阅读整个文档并提供反馈。

**当所有节都已起草和完善后：**
宣布所有节都已起草。表示打算再次审阅完整文档。

审阅整体连贯性、流畅性、完整性。

提供任何最终建议。

询问是否准备好进入读者测试，或者是否还想进一步完善。

## 第三阶段：读者测试

**目标：** 用全新的 Claude 实例（无上下文污染）测试文档，验证其对读者的实际效果。

**向用户说明：**
说明现在将测试文档是否真的对读者有效。这能发现盲点——对作者来说很清楚但可能让其他人困惑的内容。

### 测试方法

**如果可以访问子代理（例如在 Claude Code 中）：**

直接执行测试，无需用户参与。

### 步骤 1：预测读者问题

宣布打算预测读者在尝试发现此文档时可能提出的问题。

生成读者实际上会问的 5-10 个问题。

### 步骤 2：用子代理测试

宣布将用全新的 Claude 实例（无本次对话的上下文）测试这些问题。

对每个问题，用文档内容和问题调用子代理。

总结读者 Claude 对每个问题的回答是否正确。

### 步骤 3：运行附加检查

宣布将执行附加检查。

调用子代理检查歧义、错误假设、矛盾。

总结发现的任何问题。

### 步骤 4：报告和修复

如果发现问题：
报告读者 Claude 在哪些方面遇到了困难。

列出具体问题。

表示打算修复这些空缺。

返回到问题节进行完善。

---

**如果无法访问子代理（例如 claude.ai 网页界面）：**

用户需要手动进行测试。

### 步骤 1：预测读者问题

询问人们在尝试发现此文档时可能会问什么问题。他们会在 Claude.ai 中输入什么？

生成读者实际上会问的 5-10 个问题。

### 步骤 2：设置测试

提供测试说明：
1. 打开全新的 Claude 对话：https://claude.ai
2. 粘贴或分享文档内容（如果使用启用了连接器的共享文档平台，提供链接）
3. 向读者 Claude 提出生成的问题

对于每个问题，指示读者 Claude 提供：
- 答案
- 是否有任何内容模糊或不清楚
- 文档假设读者已知的知识/背景

检查读者 Claude 是否给出正确答案或误解了任何内容。

### 步骤 3：附加检查

还要询问读者 Claude：
- "此文档中哪些内容对读者可能模糊或不清楚？"
- "此文档假设读者具备哪些知识或背景？"
- "是否存在任何内部矛盾或不一致之处？"

### 步骤 4：根据结果迭代

询问读者 Claude 哪里出错或遇到困难。表示打算修复这些空缺。

返回到问题节进行完善。

---

### 退出条件（两种方法适用）

当读者 Claude 能够一致正确地回答问题且不再提出新的空缺或歧义时，文档即可完成。

## 最终审查

读者测试通过后：
宣布文档已通过读者 Claude 测试。在完成前：

1. 建议他们自己做最终通读——他们拥有这份文档，对其质量负责
2. 建议仔细检查任何事实、链接或技术细节
3. 请他们确认文档达到了预期效果

询问是否需要再次审查，或工作是否已完成。

**如果用户需要最终审查，提供它。否则：**
宣布文档完成。提供几个最终建议：
- 考虑在附录中链接此对话，以便读者了解文档的开发过程
- 使用附录提供深度内容而不使主文档臃肿
- 随着从真实读者收到反馈，更新文档

## 有效引导的技巧

**语气：**
- 直接、程序化
- 简要说明理由（当影响用户行为时）
- 不要"推销"这个方法——只需执行它

**处理偏离：**
- 如果用户想跳过某个阶段：询问是否想跳过并自由撰写
- 如果用户似乎沮丧：承认这比预期花了更长时间。建议加快进度的方法
- 始终给用户调整流程的自主权

**背景管理：**
- 始终，如果提到的内容缺少背景，主动询问
- 不要让空缺积累——随时解决它们

**工件管理：**
- 使用 `create_file` 起草完整节
- 使用 `str_replace` 进行所有编辑
- 每次更改后提供工件链接
- 不要将工件用于头脑风暴列表——那只是对话

**质量优于速度：**
- 不要急于完成各阶段
- 每次迭代都应带来有意义的改进
- 目标是一份真正对读者有效的文档
